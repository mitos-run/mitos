**Goal:** Complete issue #18's threat-model requirement: "Threat model re-derived: escaping the guest into an unprivileged device-plugin-scoped stub pod must be provably better than the old privileged forkd." The existing docs/threat-model.md section 0 records the default-surface change and promises the FULL re-derivation as a later slice; this is that slice. It comprehensively enumerates the NEW per-sandbox execution surface (the husk pod), analyzes each attack surface, argues per-surface why a guest escape into the unprivileged husk-stub pod has a provably smaller blast radius than the old privileged forkd fork, states honest residuals, and keeps forkd-the-builder's (still-privileged) surface scoped to the control-plane build path. No product code; docs only; security-critical so the claims are reviewed.

**Honesty constraint:** the re-derivation does NOT overclaim. Where the husk model is strictly better (unprivileged vs privileged, dropped caps, no broad host mounts, pod netns), we say so with the specific mechanism (the CI-proven PSA-restricted-minus-2-exceptions, the dropped caps, the read-only-only snapshot mount, the mTLS-gated control channel). Where a residual remains, we state it plainly: the read-only snapshot hostPath is shared across husk pods on a node (read-only, content-addressed, integrity-verified base image, not tenant data); /dev/kvm is exposed to the pod (a KVM/kernel escape is still a host-escape, the device plugin removes the privileged requirement but not the /dev/kvm attack surface); the per-template encryption key sits in node memory while a container is open (#31 residual); forkd-the-builder remains privileged (a smaller, control-plane-only surface, not the per-sandbox run path); the device-plugin DaemonSet runs root-but-unprivileged with the kubelet-dir + read-only /dev mounts (its own surface). We do not claim "provably better" globally where it is only better on specific axes; we argue it surface by surface and tally honestly.

  - GUEST -> HUSK-STUB CONTAINER escape: a guest that breaks out of the microVM (a Firecracker/KVM escape) lands in the husk-stub container. Compare to the old forkd fork: the husk container is unprivileged (privileged:false), all Linux capabilities dropped, allowPrivilegeEscalation:false, seccomp RuntimeDefault, runs the PSA-restricted controls (CI-proven, slice 4) except the 2 documented exceptions; the only host mount is the READ-ONLY snapshot dir; the only device is /dev/kvm (via the plugin). The old forkd was a ROOT process with /dev/kvm + a broad capability set + host access. So the post-escape blast radius is strictly smaller: no root, no caps, no broad host FS, only a read-only base-image mount + the pod netns. State this is the core "provably better" claim and exactly which controls back it.
  - The CONTROL CHANNEL (mTLS): activating a husk pod delivers tenant secrets + the enc key; the channel is mTLS, authorized to the controller identity only (slice 2, CI-proven a wrong-CA activate is rejected); an in-cluster actor cannot activate/hijack a husk pod or inject secrets. Residual: a compromised controller can activate any husk pod (the controller is the trust anchor, same as before).
  - The READ-ONLY SNAPSHOT HOSTPATH: the node template snapshot is mounted read-only; it is content-addressed, integrity-verified on load (#9), and version-compat-checked (#32). Residual: all husk pods on a node share the same read-only snapshot dir (a base image, not tenant data); a husk pod cannot WRITE it; cross-pod isolation of the mount is read-only. State honestly this is a shared read-only host mount (an exception to PSA restricted) and why it is acceptable (read-only, integrity-verified, non-tenant base image).
  - The DEVICE /dev/kvm: injected by the device plugin (no privileged, no raw hostPath device mount). Residual: /dev/kvm is exposed to the pod, so a KVM/kernel escape is still a host-escape vector (the device plugin removes the privileged requirement, NOT the /dev/kvm attack surface itself; this is inherent to any Firecracker host). The device-plugin DaemonSet itself runs root-but-unprivileged (caps dropped, the kubelet-dir + read-only /dev mounts) - its own small surface.
  - The POD NETNS: the husk pod's egress (the VM tap in the pod netns) is governed by Kubernetes NetworkPolicy/Cilium (slice 4); the bespoke host-nftables engine governs the raw-forkd mode. The per-mode enforcement (not both) is stated. Residual: in-VM enforcement of NetworkPolicy on the VM tap is proven only on a KVM-capable kubelet (bare metal).
  - The IN-POD SANDBOX API: exec/files are gated by the per-sandbox bearer token (slice 2, CI-proven untokened/wrong-token rejected). 
  - The ENC KEY (#31 PR2): the per-template key travels only over the mTLS control channel, held in node memory while a container is open, never on the node disk. Residual: the in-memory key window (a node-memory dump while a container is open yields the key; HSM offload is the follow-up).
  - The EVICTION/DRAIN surface (slice 4b): a PDB bounds disruption; a lost husk pod re-pends the claim; the drain policy governs an active sandbox. No new escape surface; a note on availability vs the old model.
  - A TALLY: a per-axis comparison table (privilege, capabilities, host FS access, device access, network governance, secret delivery) old-forkd vs husk-pod, showing the husk model is better on the privilege/caps/host-FS/network axes and equal on the inherent /dev/kvm-and-kernel axis, with the residuals named.
